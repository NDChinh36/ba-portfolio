## 1. Tổng quan module

### 1.1 Phạm vi

| Trong scope | Ngoài scope |
|---|---|
| P2P cá nhân → cá nhân qua SĐT | Business disbursement (sản phẩm riêng) |
| P2P qua QR cá nhân | Thanh toán merchant QR (Module 06) |
| Nhận tiền bằng cách hiển thị QR cá nhân | Chuyển tiền xuyên biên giới / ngoại tệ |
| Xem lịch sử chuyển tiền | Hoàn tiền (reversal) tự động |

### 1.2 Quy tắc cốt lõi

<Warning>
  **Giao dịch tức thì — Không thể hoàn:** Sau khi Sender nhấn "Xác nhận", hệ thống thực hiện debit/credit ngay lập tức. Không có cơ chế hủy hay hoàn tiền tự động. Tranh chấp được xử lý bởi CS Team.
</Warning>

### 1.3 Hạn mức theo KYC Tier

| Hạn mức | Tier 1 | Tier 2 | Tier 3 |
|---|---|---|---|
| Chuyển / lần | ❌ | 20,000,000 VND | 100,000,000 VND |
| Chuyển / ngày | ❌ | 100,000,000 VND | 500,000,000 VND |
| Chuyển / tháng | ❌ | 100,000,000 VND | 500,000,000 VND |
| Nhận / lần | Không giới hạn | Không giới hạn | Không giới hạn |
| Số tiền tối thiểu | — | 1,000 VND | 1,000 VND |

### 1.4 Authentication

| Điều kiện | Auth yêu cầu |
|---|---|
| Mọi lệnh chuyển tiền P2P | **PIN bắt buộc** (single factor, không phân biệt số tiền) |

## 3. User Flow — P2P qua SĐT

```mermaid
flowchart TD
    A([User nhấn\n"Chuyển tiền"]) --> B{Tier ≥ 2?}
    B -- Tier 1 --> C[Banner: Cần KYC\ntrước khi chuyển tiền]
    B -- Tier 2+ --> D[SCR-TRANS-01\nNhập SĐT người nhận]

    D --> E[Nhập SĐT\n10 chữ số]
    E --> F{Validate SĐT}
    F -- Sai format --> G[Inline error]
    G --> E
    F -- Hợp lệ --> H[API: Tìm tài khoản]

    H --> I{Tìm thấy?}
    I -- Không --> J[Error: SĐT chưa\nđăng ký ví]
    I -- Bị suspended --> K[Error: Không thể\nchuyển đến TK này]
    I -- Chính mình --> L[Error: Không thể\nchuyển cho chính mình]
    I -- Tìm thấy --> M[SCR-TRANS-02\nHiển thị tên + avatar]

    M --> N{User xác nhận\nđúng người?}
    N -- Sai người --> D
    N -- Đúng --> O[SCR-TRANS-03\nNhập số tiền]

    O --> P[Nhập số tiền\n+ nội dung tùy chọn]
    P --> Q{Validate\nrealtime}
    Q -- Lỗi --> P
    Q -- OK --> R[Limit Engine check]
    R --> S{Vượt hạn mức?}
    S -- Có --> T[Hiển thị lỗi\nhạn mức cụ thể]
    S -- Không --> U[SCR-TRANS-04\nConfirm screen]

    U --> V[SCR-TRANS-05\nNhập PIN]
    V --> W{PIN đúng?}
    W -- Sai --> X[Lỗi + số lần còn lại]
    X --> V
    W -- Đúng --> Y[SCR-TRANS-06\nĐang xử lý]

    Y --> Z[Atomic:\nDebit Sender ← Credit Recipient]
    Z --> AA{Kết quả}
    AA -- Thành công --> AB[SCR-TRANS-07\nThành công]
    AA -- Balance lỗi\nrace condition --> AC[SCR-TRANS-08\nThất bại: Số dư không đủ]
    AA -- Timeout > 30s --> AD[SCR-TRANS-08\nĐang xử lý — kiểm tra lịch sử]
```

## 6. Sequence Diagram — Transfer P2P (End-to-End)

```mermaid
sequenceDiagram
    autonumber
    actor Sender
    participant App
    participant TransferSvc as Transfer Service
    participant UserSvc as User Service
    participant LimitEngine
    participant WalletCore as Wallet Core
    participant NotifSvc as Notification Service
    actor Recipient

    Sender->>App: Nhập SĐT + số tiền + PIN
    App->>TransferSvc: POST /transfer/p2p\n{sender_id, recipient_phone,\namount, description, idempotency_key, pin_hash}

    TransferSvc->>TransferSvc: Verify PIN hash
    TransferSvc->>UserSvc: GET /users/lookup?phone={recipient_phone}
    UserSvc-->>TransferSvc: {user_id, wallet_id, full_name, avatar, account_status}

    Note over TransferSvc: Kiểm tra: recipient active, sender ≠ recipient\nUserService trả wallet_id cùng với user_id để\nTransferSvc gọi AtomicTransfer mà không cần lookup thêm

    TransferSvc->>LimitEngine: CheckLimit(sender_id, amount, "TRANSFER")
    LimitEngine-->>TransferSvc: {allowed: true, remaining_daily, remaining_monthly}

    TransferSvc->>WalletCore: POST /internal/wallet/atomic-transfer\n{sender_wallet_id, recipient_wallet_id, amount, idempotency_key}

    Note over WalletCore: BEGIN TRANSACTION\nDebit sender wallet\nCredit recipient wallet\nCOMMIT (hoặc ROLLBACK nếu fail)

    WalletCore-->>TransferSvc: {tx_id, status: "COMPLETED",\nsender_new_balance, timestamp}

    TransferSvc->>LimitEngine: UpdateUsage(sender_id, amount, "TRANSFER")

    par Notifications (async)
        TransferSvc->>NotifSvc: Push(sender_id, "TRANSFER_SENT", {amount, recipient_name, tx_id})
        TransferSvc->>NotifSvc: Push(recipient_id, "TRANSFER_RECEIVED", {amount, sender_name, tx_id})
    end

    TransferSvc-->>App: 200 {tx_id, status, new_balance, timestamp}
    App->>Sender: SCR-TRANS-07: Chuyển tiền thành công!
    NotifSvc-->>Recipient: Push: "Nhận {amount} đ từ {sender_name}"
```

## 8. Validation Rules

| Field / Bước | Rule | Thông báo | Trigger |
|---|---|---|---|
| **SĐT người nhận** | Bắt buộc | "Vui lòng nhập số điện thoại" | On submit |
| | 10 chữ số, đầu số hợp lệ VN | "Số điện thoại không hợp lệ" | On blur |
| | Tài khoản tồn tại | "Số điện thoại này chưa đăng ký ví" | After API |
| | Tài khoản active | "Không thể chuyển đến tài khoản này lúc này" | After API (không nêu lý do cụ thể) |
| | Không được là chính sender | "Không thể chuyển tiền cho chính mình" | After API |
| **Số tiền** | ≥ 1,000 VND | "Số tiền tối thiểu là 1,000 đ" | On blur |
| | ≤ 20,000,000 VND / lần (Tier 2) | "Số tiền tối đa 1 lần chuyển là 20,000,000 đ" | On blur |
| | ≤ `balance_available` | "Số dư khả dụng không đủ. Hiện có {X} đ" | On blur + API check |
| | ≤ Hạn mức ngày còn lại | "Vượt hạn mức ngày. Còn {X} đ hôm nay." | After API |
| | ≤ Hạn mức tháng còn lại | "Vượt hạn mức tháng. Còn {Y} đ tháng này." | After API |

## 9. Business Rules

| ID | Rule | Áp dụng tại |
|---|---|---|
| BR-TRANS-01 | Tier 1 không được chuyển tiền (chỉ nhận) | SCR-TRANS-01 gate, API |
| BR-TRANS-02 | Số tiền tối thiểu: 1,000 VND | Validation |
| BR-TRANS-03 | Phải xác nhận tên người nhận trước khi tiếp tục (mandatory confirm screen) | SCR-TRANS-02 |
| BR-TRANS-04 | Giao dịch tức thì, không thể hoàn sau khi confirm | Transfer Service |
| BR-TRANS-05 | PIN bắt buộc cho mọi lệnh chuyển, bất kể số tiền. PIN được verify server-side qua Auth Service (bcrypt compare); Transfer Service không tự validate PIN mà gọi Auth Service — xem PRD-EW-AUTH-001 BR-AUTH-06 cho lockout policy (5 sai → 30 phút) | SCR-TRANS-05, Auth Service |
| BR-TRANS-06 | Idempotency key bắt buộc — unique per sender per 24 giờ | Transfer Service |
| BR-TRANS-07 | Atomic debit/credit: cả hai xảy ra cùng lúc hoặc không xảy ra | Wallet Core (DB transaction) |
| BR-TRANS-08 | Timeout Wallet Core > 30 giây: không retry tự động; trả PENDING status | Transfer Service |
| BR-TRANS-09 | Nếu status = PENDING sau timeout: log; kiểm tra bằng idempotency khi user mở lại | Idempotency layer |
| BR-TRANS-10 | Hạn mức ngày reset 00:00 UTC+7; hạn mức tháng reset ngày 1 UTC+7. **Hạn mức TRANSFER và QR_PAY dùng chung 1 bucket "DEBIT"** — cộng gộp vào nhau; xem BR-WALLET-14 và BR-QR-08 | Limit Engine |
| BR-TRANS-11 | Thông tin recipient hiển thị: chỉ tên đầy đủ và avatar — không lộ SĐT đầy đủ, email | Privacy |
| BR-TRANS-12 | Log đầy đủ mọi GD: sender_id, recipient_id, amount, tx_id, auth_level, ip, device_id, timestamp | Audit Logger |
| BR-TRANS-13 | Rate limit chuyển tiền: tối đa 5 GD/phút/user (sliding window tại API Gateway); vượt ngưỡng → 429 TRANS_013; không tính GD failed hoặc timeout vào counter | API Gateway, LimitEngine |

## 13. Edge Cases

| Trường hợp | Xử lý |
|---|---|
| Sender mất mạng sau PIN đúng, trước khi nhận response | Khi reconnect: App gọi `GET /transfer/status?idempotency_key=`; nếu COMPLETED → show success; nếu không tìm thấy → FAILED (không retry) |
| Sender và Recipient gửi tiền cho nhau cùng lúc | Mỗi GD độc lập; Wallet Core xử lý tuần tự; không deadlock (dùng row-level lock theo thứ tự wallet_id nhỏ → lớn) |
| Balance đủ khi check (bước limit) nhưng không đủ khi Wallet Core debit (race condition) | Wallet Core trả fail → Transfer Service retry 0 lần (no retry) → trả TRANS_006 cho user |
| Recipient tài khoản bị suspend đúng lúc nhận | Trường hợp hiếm: GD vẫn credit vào ví (ví hoạt động độc lập với account status); nhưng Recipient không dùng được tiền cho đến khi tài khoản được mở lại |
| Notification service down khi GD hoàn thành | GD vẫn COMPLETED và valid; retry notification async tối đa 3 lần; không rollback GD |
| User nhập SĐT người thân rồi transfer nhiều lần nhanh (rapid fire) | Rate limit: tối đa 5 GD / phút / user; quá ngưỡng → trả 429 "Bạn đang thực hiện quá nhiều giao dịch. Vui lòng thử lại sau vài giây." |
| QR cá nhân có amount bị chụp lại và dùng lại | Dynamic QR với amount expire sau 10 phút (server-side check khi scan); static QR không expire nhưng không có amount embed |
| Transfer khởi tạo lúc 23:59, daily limit check pass, DB commit lúc 00:01 sang ngày mới | Daily limit tính theo thời điểm **LimitEngine deduct** (server timestamp lúc commit), không phải lúc nhập số tiền; GD này tính vào ngày mới — limit của ngày mới áp dụng |
| Recipient vừa bị suspend đúng lúc GD đang atomic debit/credit | Transfer Service kiểm tra recipient status trước khi bắt đầu DB transaction; nếu suspend phát hiện trong transaction → rollback; trả TRANS_002. Race condition cực hiếm — chấp nhận bằng retry logic phía người dùng |

<AccordionGroup>
  <Accordion title="[Sprint 1] Rate Limit theo Tier — Tier 3 cao hơn 5 GD/phút" icon="sliders">
    **Team:** Risk + Product · **Ưu tiên:** High

    Nâng cấp rate limit chuyển tiền từ **flat 5 GD/phút** thành **rate limit theo tier KYC**: Tier 1 = 3 GD/phút, Tier 2 = 5 GD/phút, Tier 3 = 10 GD/phút. Phục vụ nhu cầu hợp lệ của user đã KYC đầy đủ mà không gây friction không cần thiết.

    **Yêu cầu chính:** Risk Team phê duyệt ngưỡng từng tier; LimitEngine đọc tier từ user_profile; cập nhật BR-TRANS-06 và error message hiển thị đúng limit theo tier của user.  
    **Phụ thuộc:** Wallet Core LimitEngine hỗ trợ dynamic limit config per tier; KYC module expose tier field qua API.
  </Accordion>

  <Accordion title="[Sprint 1] Auto-Resolve GD PENDING — Timeout Policy" icon="clock-rotate-left">
    **Team:** Tech Lead · **Ưu tiên:** High

    Xác định và implement **thời gian tự động resolve GD PENDING**: đề xuất 1 giờ cho lỗi kỹ thuật nội bộ, 24 giờ cho GD đang chờ phía đối tác. Hiện tại PENDING có thể tồn tại vô thời hạn gây rủi ro số dư bị treo.

    **Yêu cầu chính:** Background job scan và resolve expired PENDING transactions; cập nhật ledger Wallet Core; gửi notification kết quả (COMPLETED hoặc FAILED) cho cả Sender và Recipient.  
    **Phụ thuộc:** Risk Team xác nhận timeout threshold cho từng loại PENDING; CS xây dựng SOP cho case cần manual resolve.
  </Accordion>

  <Accordion title="[Sprint 2] Recent Recipients — Số lượng & TTL lưu trữ" icon="users">
    **Team:** Product · **Ưu tiên:** Medium

    Xác định policy lưu **lịch sử người nhận gần đây** trên SCR-TRANS-01: số lượng tối đa (đề xuất 10 người) và thời gian lưu (đề xuất 90 ngày kể từ GD gần nhất). Giúp user gửi tiền cho người thường xuyên giao dịch nhanh hơn.

    **Yêu cầu chính:** API `GET /transfers/recipients/recent`; UI avatar + tên + SĐT rút gọn trên horizontal scroll; xóa cache khi user xóa GD history.  
    **Phụ thuộc:** Privacy review — cần user consent cho việc lưu lịch sử người nhận theo Nghị định 13/2023.
  </Accordion>

  <Accordion title="[Sprint 2] Favorite Contacts — Đánh dấu thủ công hoặc tự động từ tần suất" icon="star">
    **Team:** Product · **Ưu tiên:** Low

    Tính năng **Người nhận yêu thích**: user chủ động đánh dấu (star icon) hoặc hệ thống tự động gợi ý dựa trên tần suất (ví dụ: người nhận ≥ 3 GD/tháng). Hiển thị ưu tiên trên SCR-TRANS-01 phía trên Recent list.

    **Team:** Product · **Ưu tiên:** Medium

    **Yêu cầu chính:** NotifService kiểm tra `push_enabled` của Recipient trước khi gửi; fallback SMS nếu push disabled; template SMS ngắn gọn cho GD nhận tiền.  
    **Phụ thuộc:** OTP Gateway hỗ trợ transactional SMS (khác với OTP SMS); ước tính chi phí SMS per notification và ngưỡng kích hoạt.
  </Accordion>

## 15. Non-Functional Requirements (NFR)

### Performance

| Chỉ số | Target | Ghi chú |
|---|---|---|
| User lookup (tìm người nhận theo SĐT) | P95 < 300ms | Index trên phone field |
| Transfer API (end-to-end) | P95 < 3s | Từ PIN submit đến COMPLETED response |
| Atomic debit/credit (DB transaction) | P95 < 500ms | PostgreSQL serializable transaction |
| Limit check (LimitEngine) | P95 < 100ms | Redis cache cho daily/monthly counters |
| Notification dispatch (async) | P95 < 5s | Không block transfer response |
| Transfer history load | P95 < 500ms | Paginated; không full table scan |

### Bảo mật (Security)

| Yêu cầu | Mô tả |
|---|---|
| PIN verification | Server-side bcrypt compare trước mỗi transfer; không trust client-side PIN check |
| Idempotency key | UUID v4 unique per sender per 24h; server cache 24h để chống duplicate GD |
| Rate limiting | Tối đa 5 GD/phút/user; implement với sliding window algorithm tại API Gateway |
| Audit trail | Mọi GD ghi: sender_id, recipient_id, amount, auth_level, ip, device_id, idempotency_key, timestamp |
| Recipient info masking | Chỉ trả về tên đầy đủ + avatar; không lộ SĐT đầy đủ, email, địa chỉ trong API response |
| Row-level locking | Lock wallet rows theo thứ tự wallet_id nhỏ → lớn để tránh deadlock trong concurrent transfers |

### Tính sẵn sàng & Độ tin cậy

| Chỉ số | Target | Ghi chú |
|---|---|---|
| Transfer Service uptime | ≥ 99.9% | Payment-critical |
| Wallet Core dependency | ≥ 99.95% | Transfer phụ thuộc Wallet Core; SLA phải cao hơn |
| Idempotency cache durability | Redis Cluster với AOF persistence | Không mất cache khi Redis restart |
| Timeout handling | 30s timeout → trả PENDING, không retry | Idempotency key cho phép safe re-check |
| Notification service failure | GD vẫn COMPLETED; retry notification async 3 lần | Không rollback GD vì notification fail |

### Lưu trữ & Compliance

| Loại dữ liệu | Thời gian lưu | Cơ sở pháp lý |
|---|---|---|
| Transfer records (mọi GD) | 5 năm | Nghị định 52/2024 |
| Transfer audit logs (auth, device, IP) | 5 năm | AML/fraud investigation |
| Idempotency key cache | 24 giờ (rolling) | Chống duplicate trong window |
| Limit usage counters (daily/monthly) | 13 tháng | Dispute resolution; chứng minh limit đúng |
| PENDING transaction records | Vĩnh viễn cho đến khi resolve | Không được xóa GD PENDING |
